git规范

git规范

目前项目分为4个环境

  • dev 【前端专用地址】
  • qa 【测试专用地址】
  • pre 【预发布环境===生产环境（正式环境）】
  • master【生产环境，开发推了master（不会生效），需要通知管理人员手动推上正式环境】
  • 本地开发分支【使用开发者名：zhangshan，若重名：zhangshan_xxx】
  • ==========================================================
  • sprint分支 【发布和QA测试，该分支对应到QA站和预发布站 】
  • feature分支 【新功能开发，从主库fork到自己的分支，从当前sprint分支拉出feature分支进行新功能的开发，该分支对应本地或DEV环境】
    • feature分支命名为 story[TAPD编号]-[功能简称]，例如: story11250-login-improve.
  • hotfix分支 【紧急bug处理】
    • 命名为 bug[BUG编号]-[问题简称]，例如 bug1123-fix-price-display

代码提交规范

  • 提交前review代码
  • merge 前/pull 前，先提交本地代码
  • 填写有意义的 commit message
    • feat: 一个新功能
    • fix: 一个 bug 修复，复制 bugtags 上的bug标题
    • docs: 仅仅修改了文档，比如 README, CHANGELOG, CONTRIBUTE 等
    • format: 不影响代码逻辑的修改，比如空格、格式缩进、删除分号等
    • refactor: 代码重构
    • perf: 提升性能的改动
    • test: 增加或修改测试
    • chore: 改变构建流程、或者增加辅助工具、依赖库等
  • 修复一个bug，提交一次

发布

最好有个tag标记版本，如 v1.1.0
